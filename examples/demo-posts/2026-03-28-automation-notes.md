자동화는 한 번 넣으면 계속 남습니다. 그래서 처음부터 복잡하게 만들기보다, 사람이 읽기 쉬운 단계로 유지하는 것이 더 중요합니다.

## 설치 단계는 의존성을 숨기지 않는다

파이프라인 초반부에서 무엇이 설치되는지 명확하게 보여야 합니다.

```bash
bundle install
bundle exec jekyll build
```

명령이 단순하면 환경이 달라져도 문제를 찾기 쉽습니다. 반대로 너무 많은 래퍼 스크립트 안에 숨기면 실패 지점을 찾는 시간이 길어집니다.

## 산출물 검증은 꼭 넣는다

빌드가 성공했다고 바로 배포하면 빈 디렉터리나 잘못된 경로가 그대로 올라갈 수 있습니다.

그래서 최소한 아래 두 검사는 항상 넣는 편입니다.

- 결과 디렉터리가 존재하는가
- 대표 HTML 파일이 실제로 생성됐는가

작아 보이지만 이 두 줄이 많은 실수를 막아줍니다.

## 배포 브랜치는 목적을 하나만 가진다

`gh-pages` 브랜치에는 빌드 결과만 있고, 소스는 `main`에만 있도록 두면 복구가 쉬워집니다.

이 구조의 장점은 명확합니다.

- 현재 배포 중인 파일을 바로 확인할 수 있다
- 이전 상태로 되돌리기 쉽다
- 정적 호스팅 서비스와 연결이 단순하다

## 끝으로

자동화는 화려한 기능보다 **읽기 쉬운 단계와 분명한 실패 지점**이 더 중요합니다. 블로그처럼 자주 고치는 프로젝트일수록 단순한 파이프라인이 오래 살아남습니다.
