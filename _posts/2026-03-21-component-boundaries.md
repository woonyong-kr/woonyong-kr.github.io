컴포넌트를 나눌 때 흔히 `재사용 가능할까?`를 먼저 묻습니다. 물론 중요하지만, 실제로 더 오래 남는 기준은 아래 세 가지였습니다.

## 1. 누가 상태를 소유하는가

상태가 자주 바뀌는 화면일수록 컴포넌트 경계는 마크업보다 상태 소유권에 맞춰야 합니다.

예를 들어 글 목록 카드에서 다음 상태가 섞이면 금방 복잡해집니다.

- 정렬 기준
- 필터 조건
- 현재 페이지
- 로딩 상태
- 빈 상태

이때 카드 자체는 최대한 읽기 전용으로 두고, 목록 컨테이너가 상태를 소유하게 두는 편이 낫습니다.

## 2. 변경 이유가 같은가

같은 화면 안에 있어도 변경 이유가 다르면 분리해야 합니다.

예를 들어 프로필 헤더와 글 탭은 가까이 붙어 있지만 바뀌는 이유가 다릅니다.

- 프로필 헤더는 사용자 정보가 바뀔 때 수정된다
- 글 탭은 탐색 구조가 바뀔 때 수정된다

이 둘을 한 파일에서 같이 다루기 시작하면 작은 수정도 불필요하게 넓어집니다.

## 3. 스타일 토큰을 공유하는가

디자인을 비슷하게 맞추는 것과 같은 컴포넌트인 것은 다릅니다. 간격, 색상, 타이포 토큰만 공유하고 실제 구조는 분리하는 편이 더 자주 이깁니다.

```html
<section class="profile"></section>
<nav class="velog-tabs"></nav>
<article class="post-card"></article>
```

이렇게 역할이 분명하면 나중에 일부만 교체해도 전체 인상이 크게 무너지지 않습니다.

## 결론

컴포넌트 경계는 보기 좋은 파일 분할보다 **상태, 변경 이유, 토큰 공유 여부**로 정하는 편이 오래 갑니다. 화면 구조가 비슷해 보여도 책임이 다르면 분리하고, 책임이 같으면 파일 수가 조금 적어도 괜찮습니다.
